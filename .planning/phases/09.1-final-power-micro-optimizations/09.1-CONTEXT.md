# Phase 9.1: Final Power Micro-Optimizations - Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Four zero-risk micro-optimizations: (1) Reset CPU to 80MHz after WiFi disconnect before LED feedback, (2) Reduce serial flush delay 100ms→20ms, (3) Hold NeoPixel GPIO 27 LOW during deep sleep to cut WS2812 quiescent current, (4) Reduce API retry delay 500ms→300ms.

</domain>

<decisions>
## Implementation Decisions

### CPU Frequency Reset
- Add `set_cpu_freq(80)` after WiFi disconnect (line ~807) and before LED feedback
- LED feedback runs at 80MHz instead of 160MHz — saves ~15mA during ~800ms of blinks

### Serial Flush Delay
- Change `time.sleep_ms(100)` to `time.sleep_ms(20)` in `enter_deep_sleep()`
- 20ms is more than enough for 115200 baud UART flush

### NeoPixel GPIO Hold
- After `led.off()`, before `deepsleep()`, use `machine.Pin(27).init(machine.Pin.OUT)` then `machine.Pin(27).value(0)` to ensure LOW
- If available, use `esp32.gpio_hold_en(27)` to keep GPIO 27 LOW during deep sleep (prevents NeoPixel controller from drawing quiescent current)
- Wrap in try/except — not all MicroPython builds support `gpio_hold_en`

### API Retry Delay
- Change `time.sleep_ms(500)` to `time.sleep_ms(300)` in `send_command()` retry loop
- 300ms is still sufficient for transient network recovery

### Claude's Discretion
- Exact implementation of GPIO hold (api varies by MicroPython version)
- Whether to add `gpio_deep_sleep_hold_en()` for all held pins
- Test strategy for these changes

</decisions>
